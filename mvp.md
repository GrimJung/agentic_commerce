# MVP 문서 (H-AI 여행 예약 챗봇)

본 문서는 **단계형 프로덕트 기획 인터뷰 프롬프트**(Step 0~9)의 최종 정리 형식인 **기획 요약서** 구조를 따릅니다. H-AI 프로젝트에 맞춰 채운 내용과, MVP 범위의 입력·출력·성공 기준·예외·사용자 상세를 담습니다.

---

## 1. 배경

- **시작 계기:** 사용자와 대화하며 패키지·자유여행(FIT)을 추천하고 예약·결제까지 이어지는 대화형 AI 커머스 경험을 제공하려는 기획. 기존 검색형 예약과 달리 통합 검색·비교·예약 플로우와 AI 추론 시각화로 차별화.
- **인식한 문제:** 검색·비교·예약이 여러 단계·화면에 분리되어 있고, 추천 이유가 불투명하며, 패키지와 FIT를 한곳에서 유연하게 검색하기 어렵다.
- **타이밍:** AI 기반 추천·대화형 UX 확대, 패키지·FIT 통합 검색 수요, 추천 근거 투명화(추론 과정 노출)에 대한 관심 증가.

---

## 2. 목적

- **비전 (3년):** 여행 검색·추천·예약을 대화 한 흐름으로 처리하는 대표 AI 커머스 경험 제공. 실제 항공/숙소 API·결제 연동과 개인화·재구매 경험 강화.
- **미션:** 사용자가 “왜 이 상품이 추천되었는지” 이해하며, 패키지와 FIT를 한 플로우에서 검색·비교·예약할 수 있게 한다.
- **6개월 목표:** MVP 플로우 검증·고도화. 랜딩 → 채팅 → 선호도 → 추천·비교·예약·결제·확정까지 한 번에 완주 가능한 상태.
- **핵심 지표:** (1) 랜딩→채팅 진입률, (2) 선호도 제출 후 추천 결과 노출률, (3) 예약 플로우 완주율(예약 확정까지).

---

## 3. 타겟 사용자

- **주 타겟:** 여행 상품을 검색·비교·예약하려는 일반 고객. 패키지 선호 사용자와 FIT(항공·숙소 조합/단독) 선호 사용자로 세분. 페르소나(이전 검색·선호)가 있는 재방문 사용자는 Smart-Fill로 개인화 제공.
- **절실함·지불 의향:** 검색·비교·예약 통합과 추천 근거 투명화에 대한 니즈 존재. 지불 의향·가격대는 시장 조사·검증 필요.
- **접근 방법:** 웹 앱(모바일 우선), 향후 채널 확장 검토.

---

## 4. 문제 정의

- **핵심 문제:** 여행 예약 시 검색·비교·예약이 분리되어 있고, 추천 이유가 불투명하며, 패키지와 FIT를 한곳에서 유연하게 다루기 어렵다.
- **영향 받는 사람:** 패키지 또는 FIT로 여행을 찾는 모든 예약 고객.
- **현재 해결 방법:** 기존 OTA·메타검색·챗봇 사용. 대화형 통합 플로우와 추론 시각화는 부족.

---

## 5. 솔루션

- **한 줄 설명:** “여행 패키지·FIT를 대화로 검색·비교·예약할 수 있는 AI 챗봇”
- **핵심 차별점:** 대화형 UX + AI 추론 과정 시각화(AgentReasoningBlock) + 패키지·FIT(조합/항공만/숙소만) 통합 검색·예약.
- **가치 제안:** 자연어/선호도 기반 맞춤 추천, “왜 이 상품인지” 단계별 노출로 신뢰·이해도 제고, 한 플로우에서 패키지·FIT 모두 처리.

---

## 6. 핵심 기능

### 6.1 MVP 필수 기능

- 대화형 진입(랜딩 → “상품 추천받고 결제하기” → 채팅, “상품 추천해줘” 자동 전송)
- 여행 타입·선호도 입력(패키지/FIT, 목적지·테마·예산, FIT 시 검색 모드, Smart-Fill)
- AI 추론 시각화(검색·비교 시 단계별 진행 메시지)
- 통합 추천·탐색(패키지/FIT 조합·항공·숙소 카드, 상세·비교·재추천)
- 운임·룸 타입 선택(FIT·숙소 예약 시)
- 예약·결제·확정(BookingForm → PaymentModal → BookingConfirmation)
- 취소·복귀(결제 취소, 헤더 뒤로가기 → 랜딩)

### 6.2 사용자 플로우

```
랜딩 → [상품 추천받고 결제하기] → 채팅(자동 "상품 추천해줘")
  → 여행 타입 선택(패키지/FIT) → 선호도 입력
  → AI 검색(추론 시각화) → 추천 카드(최대 3개)
  → 상세/비교/재추천 반복 → [예약하기]
  → (해당 시) 운임·룸 타입 선택 → 예약자 정보 → 결제 → 예약 확정
```

### 6.3 어떤 입력을 받는지

| 구분 | 입력 항목 | 필수 여부 |
|------|------------|-----------|
| 진입 | "상품 추천받고 결제하기" 클릭 | 필수 |
| 채팅 | 자연어 메시지(의도·Smart-Fill 활용) | 선택 |
| 여행 타입 | 패키지 상품 / 자유여행(FIT) | 필수 |
| 페르소나 응답 | [추천 검색하기] / [직접 검색하기] | 선택 |
| 선호도 | 목적지, 테마(패키지), 예산, 여행자 수, FIT 시 검색 모드(조합/항공만/숙소만) | 조건부 필수 |
| 결과 탐색 | 카드 클릭, 비교하기, 추천 다시받기 | 선택 |
| 예약 | 예약하기, 운임 선택, 룸 타입 선택 | 해당 플로우 시 |
| 예약자 정보 | 이름, 연락처, 이메일, 생년월일, 여권번호, 여행자 수, 약관 동의 | 필수 |
| 결제 | 결제 방법 선택 및 결제 실행 | 결제 시 필수 |

### 6.4 어떤 출력 구조인지

- **추천 결과:** PackageData[] / FITPackageData[] / FlightData[] / HotelData[] (각 최대 3개)
- **상세:** PackageDetail, FITPackageDetail, FlightDetail, HotelDetail(바텀시트)
- **비교:** PackageComparison, FITComparison, FlightComparison, HotelComparison(바텀시트)
- **예약 확정:** bookingNumber(HAI+8자리), packageTitle, travelers, amount
- **기타:** AgentReasoningBlock(검색·비교 시), 채팅 메시지, FIT 완료 제안 메시지

### 6.5 성공 기준

| 번호 | 성공 기준 | 확인 방법 |
|------|-----------|------------|
| 1 | 랜딩 → 채팅 진입 | "상품 추천받고 결제하기" 클릭 시 채팅 전환 + "상품 추천해줘" 자동 전송 |
| 2 | 여행 타입 선택 후 선호도 입력 | 패키지/FIT 선택 시 선호도 폼 노출, 필수 입력 후 검색 실행 |
| 3 | 추천 결과 표시 | AI 추론 단계 표시 후 해당 타입 추천 카드 최대 3개 표시 |
| 4 | 상세·비교·재추천 | 카드 클릭 → 상세 바텀시트, 비교하기 → 비교 바텀시트, 추천 다시받기 → 새 카드 |
| 5 | 예약 플로우 완주 | 예약하기 → (해당 시 운임·룸 타입) → 예약자 정보 → 결제 → 예약 확정 모달 |
| 6 | 예약 확정 출력 | 예약 번호, 상품명, 여행자 수, 결제 금액 표시 |
| 7 | 취소·뒤로가기 | 결제 취소 시 이전 단계, 헤더 뒤로가기 시 랜딩 복귀 |

### 6.6 예외 상황

| 예외 상황 | 처리 방향 (MVP) |
|-----------|------------------|
| 선호도 미입력 | 패키지: theme/budget/destination 중 하나 이상 필수. FIT: budget/destination 중 하나 이상. 미충족 시 제출 불가·안내 |
| 검색 결과 없음 | 목업 기준 미발생. API 연동 시 "검색 결과 없음" 메시지·조건 완화 제안 |
| 결제 취소 | PaymentModal 닫기, 이전 단계 복귀 |
| 결제 실패 | 목업 결제. 연동 시 실패 메시지, 재시도/취소 제공 |
| 필수 예약자 정보 미입력 | BookingForm 유효성 검사, 제출 차단·필드별 오류 메시지 |
| API/백엔드 장애 | 연동 후 에러 메시지, 재시도, 폴백(캐시/목업) 정책 |
| 비교 시 상품 부족 | 비교하기 버튼 2개 이상일 때만 노출 |
| 뒤로가기 시 상태 | main(랜딩) 이동, 진행 상태 초기화 안내 |

## 7. 비즈니스 모델

- **수익 모델:** 미정. 거래 수수료·마진·구독 등 검토 중.
- **가격:** 미정. 경쟁사 가격·지불 의향 조사 후 설정.
- **경쟁 우위:** 대화형·추론 시각화·패키지·FIT 통합으로 프리미엄 경험 포지셔닝.

## 8. 실행 계획

- **일정:** 단기 MVP 플로우 검증·고도화. 장기 API·결제 연동, 개인화·재구매 강화.
- **팀·리소스:** 현재 프론트(React, TypeScript, Vite, Tailwind, Radix 등), 목업 데이터. 백엔드·API·결제·AI 연동 별도 설계·구현 필요.
- **예산:** MVP(목업)는 인프라·호스팅 위주. 실제 연동 시 API·PG·백엔드·보안 비용 검토.
- **주요 리스크:** GDS/OTA API 연동·비용, 결제·에스크로·PCI, 개인정보·규제. API 장애 시 폴백(캐시/목업) 설계 권장.
