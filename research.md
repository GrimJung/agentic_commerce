## 0. 프로덕트 시장조사 & 기획 프롬프트 빌더 개요

본 문서는 **프로덕트 시장조사 & 기획 프롬프트 빌더** 구조를 따릅니다. 해당 빌더는 2단계로 진행됩니다: (1) 인터뷰를 통해 정보 수집, (2) 수집된 정보로 맞춤형 리서치 프롬프트 생성. 인터뷰 질문 흐름(Step 0, 비즈니스/기획자·PM·창업가별 A/B/C 가이드)은 별도 문서를 참조합니다.

이 문서에서는 **H-AI(하이) 여행 예약 AI 챗봇**에 대해 수집·정리된 정보를 **수집된 정보 요약**으로 담고, 이를 바탕으로 생성한 **심층 시장조사·기획용 리서치 프롬프트**를 포함합니다. 해당 프롬프트를 복사해 Claude/ChatGPT 등에 입력하면 H-AI 전용 심층 리서치를 수행할 수 있습니다.

## 1. 수집된 정보 요약

### 기본 정보

- **프로덕트:** H-AI 여행 예약 AI 챗봇 (Agentic Commerce)
- **배경:** 사용자와 대화하며 패키지·자유여행(FIT)을 추천하고 예약·결제까지 이어지는 대화형 AI 커머스 경험을 제공하려는 기획. 기존 검색형 예약과 달리 통합 검색·비교·예약 플로우와 AI 추론 시각화로 차별화.
- **현재 단계:** MVP UI 완료(React + Vite, 목업 데이터). GitHub + Vercel 배포로 라이브 데모 제공. 실제 GDS/OTA API·결제 연동은 미구현.

### 시장 분석

- **시장:** 국내·외 여행 예약·OTA·챗봇/대화형 추천 시장. 유사 서비스로 국내 항공·숙소·패키지 통합 예약 플랫폼, 대형 OTA 챗봇, 여행 스타트업 AI 추천 서비스, 해외 ChatGPT/검색 기반 여행 플래너, Kayak/Google Travel 등 메타검색·AI 여행 어시스턴트 존재.
- **트렌드:** AI 기반 추천·대화형 UX 확대, 패키지·FIT 통합 검색 수요, 추천 근거 투명화(추론 과정 노출)에 대한 관심 증가.

### 타겟 고객

- **주 타겟:** 여행 상품을 검색·비교·예약하려는 일반 고객. 패키지 선호 사용자와 FIT(항공·숙소 조합/단독) 선호 사용자로 세분. 페르소나(이전 검색·선호)가 있는 재방문 사용자는 Smart-Fill로 개인화 제공.
- **페인 포인트:** 검색·비교·예약이 여러 단계·화면에 분리되어 있음, 추천 이유가 불투명함, 패키지와 FIT를 한곳에서 유연하게 검색하기 어려움.

### 프로덕트 전략

- **차별점:** 대화형 UX + AI 추론 과정 시각화(AgentReasoningBlock) + 패키지·FIT(조합/항공만/숙소만) 통합 검색·예약.
- **핵심 기능:** 랜딩·채팅 진입, 여행 타입·선호도 입력(Smart-Fill), AI 추론 시각화, 통합 추천·탐색(상세·비교·재추천), 운임·룸 타입 선택, 예약·결제·확정, 취소·복귀.
- **MVP 범위:** 위 핵심 피처 기준으로 검증 및 고도화. 액티비티 선택 등은 선택 경로.

### 비즈니스 모델

- **수익 모델:** 미정. 거래 수수료·마진·구독 등 검토 중.
- **가격:** 미정. 경쟁사 가격·지불 의향 조사 후 설정 예정.
- **포지셔닝:** 대화형·추론 시각화·통합 플로우로 프리미엄 경험 제공 방향.

### 실행 계획

- **일정:** 단기—MVP 플로우 검증·고도화. 장기—실제 항공/숙소 API·결제 연동, 개인화·재구매 경험 강화.
- **리소스:** 현재 프론트(React, TypeScript, Vite, Tailwind, Radix 등), 목업 데이터. 백엔드·API·결제·AI 연동은 별도 설계·구현 필요.
- **리스크:** GDS/OTA API 연동·비용, 결제·에스크로·PCI 대응, 개인정보·규제. API 장애 시 폴백(캐시/목업) 설계 권장.

## 2. 상세 내용

### 2.1 제품 분석 (만들고자 하는 프로덕트 분석)

#### 제품 정의

H-AI는 여행 상품(패키지, FIT)을 대화형 UI로 검색·비교·예약할 수 있는 AI 챗봇이다. 사용자가 목적지·예산·테마 등을 입력하면 AI가 추천하고, 상세·비교·룸타입·액티비티 선택을 거쳐 예약·결제까지 한 플로우로 지원한다.

#### 주요 기능 범위

- **랜딩 및 진입:** 칩 메뉴 기반 랜딩, "상품 추천받고 결제하기"로 채팅 플로우 진입.
- **여행 타입:** 패키지 vs 자유여행(FIT). FIT 시 항공+숙소 조합 / 항공만 / 숙소만 검색 모드.
- **선호도 입력:** 목적지, 테마, 예산, 여행 시기, **인원**(패키지 직접 검색=성인/아동/유아, FIT·추천 검색=성인+아동만+아동 나이 만0~17세 선택). 페르소나 기반 Smart-Fill(자동 채움) 지원.
- **AI 검색·시각화:** 검색/비교 시 단계별 추론 메시지(AgentReasoningBlock)로 진행 상황 노출.
- **결과 탐색:** 추천 카드(최대 3개), 상세(바텀시트), 비교 테이블, 재추천.
- **예약·결제:** 운임 선택, 룸 타입 선택, 예약자 정보, 결제 모달, 예약 확정.

#### 차별화 포인트

- 대화형 UX와 단계별 AI 추론 노출로 **이해도·신뢰** 제고.
- 패키지와 FIT(조합/항공/숙소)를 하나의 플로우에서 처리하는 **통합 여행 검색·예약**.

#### 보완 필요 사항 (현재 기준)

- 상품·항공·숙소·액티비티 데이터는 목업 기준. 실제 GDS/OTA API·결제 연동은 별도 설계·구현 필요.

국내·외 여행 예약·추천 서비스 중 **챗봇/대화형 추천** 또는 **AI 기반 여행 검색**을 제공하는 사례를 대상으로 기능·UX·비즈니스 모델을 비교한다.

#### 분석 대상 (예시, 실제 서비스명으로 교체 권장)

- **국내:** 항공·숙소·패키지 통합 예약 플랫폼, 대형 OTA 챗봇, 여행 스타트업 AI 추천 서비스.
- **해외:** ChatGPT/검색 기반 여행 플래너, Kayak/Google Travel 등 메타검색, AI 여행 어시스턴트 스타트업.

#### 비교 관점 (표로 정리 권장)

| 관점       | 내용 |
| ---------- | ---- |
| 제공 형태  | 챗봇 / 검색형 / 하이브리드 |
| 지원 상품  | 패키지 / FIT / 항공·숙소 단독 |
| AI 활용    | 추천만 / 추천+예약 / 추론 과정 노출 여부 |

#### 시사점

- 경쟁사 대비 **실시간 재고·가격 API 연동**, **결제·에스크로** 구축이 과제로 도출됨.

### 2.3 기술 분석

#### 현재 기술 스택 (프로젝트 기준)

- **프론트:** React 18, TypeScript, Vite 6, Tailwind CSS 4.
- **UI:** Radix UI, MUI, Vaul(바텀시트), React Hook Form, Recharts, Motion.
- **테스트:** Playwright.
- **데이터:** 현재 목업; 실제 연동 시 REST/GraphQL API 클라이언트 및 상태 관리 확장 필요.

#### 기술 선택 이유

- **React + TypeScript:** 유지보수성, 채팅·폼·복잡 플로우에 적합.
- **Vite:** 빠른 개발 서버·빌드.
- **Tailwind + Radix:** 반응형·접근성·디자인 시스템 구축 용이.
- **단일 앱 step 기반 플로우:** main → initial → preference → packages / fit-packages → booking → payment → confirmed. 모달/바텀시트는 step 유지, UI만 전환.

#### 도입 검토 기술

- **백엔드/API:** 여행 상품·재고·가격 조회용 BFF 또는 프록시 (Node.js / 기존 백엔드 확장).
- **AI/LLM:** 추천·요약·자연어 이해용 API (OpenAI/Claude/자체 모델 등). 현재는 추론 단계 메시지 고정; 추후 동적 생성 시 토큰·지연시간 고려.
- **실시간 데이터:** GDS/OTA API, 캐싱(Redis 등), 재고·가격 TTL 정책.
- **결제:** PG사 연동(이니시스/토스/스트라이프 등), PCI DSS 대응.

#### 리스크·대응

- API 레이트 리밋·장애 시 폴백(캐시/목업) 설계.
- 결제·개인정보는 보안 정책 및 로그·감사 요구사항 정리 필요.

### 2.4 기술 비용 분석

서비스 운영에 필요한 **인프라·외부 API·인력** 관점의 비용을 구간별로 정리한다. 수치는 예시이며, 실제 견적·사용량으로 갱신해야 한다.

#### 인프라 (예시)

- **프론트 호스팅:** Vercel/Netlify/자체 웹서버. 월 $0~50 구간 가정.
- **백엔드/DB:** 필요 시 AWS/GCP 등 서버·DB 비용.
- **CDN/정적 자원:** 트래픽에 따른 비용.

#### 외부 API·SaaS

- **여행 데이터:** GDS/OTA API 사용료(건당·월정액 등).
- **AI/LLM:** OpenAI/Claude 등 호출당·토큰당 비용. 일 예상 호출량 기준 월 비용 산출.
- **결제:** PG 수수료(거래액 % + 건당).
- **모니터링/에러 추적:** Sentry 등 월 플랜.

#### 인력·개발 비용

- 프론트/백엔드/AI 연동/결제 연동 인력 및 일정.
- MVP vs 정식 오픈 단계별 비용 구간 (예: 3개월/6개월/1년).

#### 요약

- **MVP(목업 유지):** 인프라·프론트 호스팅 위주, 비용 최소.
- **실제 상품·결제 연동:** API 비용·PG 수수료·백엔드·보안 대응으로 비용 증가.
- 문서 말미에 **비용 시나리오 표**(낙관/기본/비관)를 두고, 분기별로 재검토하는 것을 권장한다.
